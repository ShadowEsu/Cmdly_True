# Play Store submission runbook

This is the lift-off checklist for getting Cmdly into Google Play once. The web build is also shipped as an installable PWA; both ship from the same `main` branch.

> Apple App Store is **explicitly out of scope for v1**. We may revisit via a Capacitor wrap later; see [Future: Capacitor](#future-capacitor).

---

## 1. Generate the release keystore (one-time, do NOT commit)

A keystore signs every release; Google Play locks your app identity to the key forever. Lose it and you cannot publish updates.

```powershell
# from the repo root, on the machine that will sign builds
keytool -genkeypair `
  -v `
  -keystore android/release.keystore `
  -alias cmdly-release `
  -keyalg RSA `
  -keysize 4096 `
  -validity 10000 `
  -storetype JKS
```

You will be prompted for:
- Keystore password (store in 1Password / Bitwarden)
- Key password (use the same value to keep things sane)
- Name / org / locality (real values; Google won't show them publicly)

After it runs, copy `android/keystore.properties.template` to `android/keystore.properties`:

```
storeFile=release.keystore
storePassword=...your store password...
keyAlias=cmdly-release
keyPassword=...your key password...
```

Both `android/release.keystore` and `android/keystore.properties` are already in `.gitignore`. They must **never** be committed, pushed, or pasted into chats. Back them up to a password manager and an offline drive — losing them ends your ability to publish updates to this listing.

Optionally, instead of `keystore.properties` you can set env vars before running `gradlew bundleRelease`:

- `CMDLY_RELEASE_STORE_FILE`
- `CMDLY_RELEASE_STORE_PASSWORD`
- `CMDLY_RELEASE_KEY_ALIAS`
- `CMDLY_RELEASE_KEY_PASSWORD`

The Gradle config picks env vars first, then `keystore.properties`, then falls back to the debug key (only for local dev).

## 2. Build a signed AAB

```powershell
cd android
./gradlew bundleRelease
```

Output: `android/app/build/outputs/bundle/release/app-release.aab`

That `.aab` is what you upload to Play Console.

## 3. Play Console one-time setup

1. Create app → Internal testing track → upload `app-release.aab`.
2. Set up:
   - **App icon**: upload `android/play-store-icon-512.png` (512×512 PNG).
   - **Feature graphic**: 1024×500 PNG. Use a clean dark gradient with the wordmark.
   - **Screenshots**: at least 4, phone size, ≥320 px on shortest edge.
   - **Short description** (≤80 chars): `Offline command reference for AI tools. Save shortcuts, browse, search.`
   - **Full description**: pulled from `marketing/video-scripts.md`. No "best", no superlatives, no platform logos.
4. **Data safety form**: select "No data collected, no data shared". Cmdly stores everything locally — there is no backend account, no analytics, no third-party SDKs in the Android build.
5. **Content rating**: Everyone (rated 3+).
6. **Target audience**: 13+ to avoid COPPA paperwork.
7. **Privacy policy URL**: paste the deployed URL of the PWA's `/privacy` page (e.g. `https://cmdly.app/privacy`). The page must be live before submission — Play crawls it.
8. **Ads**: No.
9. **In-app purchases**: No.
10. **Government / financial app**: No.

## 4. Pre-launch checks

Google's pre-launch report runs the AAB on real devices. Look at:
- Stability (no crashes on first launch, including airplane mode).
- Accessibility hints.
- Performance.

Fix anything that comes back red before promoting to closed → open → production.

## 5. Versioning

Every release must bump both:
- `versionCode` (integer, monotonically increasing) in `android/app/build.gradle.kts`
- `versionName` (semver string)

Same `applicationId` (`com.aicommandvault.app`) is locked once the app is uploaded — do not change it. The `versionName` should track the web version when the two ship in lockstep.

## 6. PWA deployment

The PWA build is what runs on the web *and* what installs from a browser on Android/desktop. To deploy:

```powershell
npm run build
```

Then upload the contents of `dist/` to any static host with HTTPS (Vercel, Cloudflare Pages, Netlify, Fly static). Required for installability:

- HTTPS
- `manifest.webmanifest` (generated by `vite-plugin-pwa`)
- `sw.js` (generated)
- `pwa-192x192.png` and `pwa-512x512.png` (generated)
- `maskable-icon-512x512.png` (generated, for adaptive home-screen icons)

`/privacy` must resolve. With React Router this works automatically as long as the host falls back to `index.html` for unknown paths (Vercel/Netlify/Cloudflare do this with a single rewrite rule).

After deploy, run Chrome's "Application → Manifest" panel on the live site; it should report "Installable: yes".

## Future: Capacitor

If/when we want a single codebase for iOS + Android instead of the standalone Kotlin app, the path is:

```powershell
npm install @capacitor/core @capacitor/cli
npx cap init Cmdly com.aicommandvault.app --web-dir dist
npx cap add android
npx cap add ios
```

That wraps the web build in a WebView and gives a native binary. It would make `android/` (the Kotlin app) obsolete. For v1 we are not doing this — the Kotlin app and the PWA both ship from `main` independently.
